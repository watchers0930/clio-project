# AI 문서 품질 검수 계획서

> **요약**: 작성/생성된 문서를 GPT-4o가 자동으로 품질 검수하고 개선 제안을 제공하는 기능
>
> **프로젝트**: CLIO
> **현재 버전**: v5.0.0
> **작성일**: 2026-04-12
> **상태**: Draft

---

## 1. 개요

### 1-1. 목적

CLIO는 AI로 공문서를 자동 생성하는 시스템이다. 그러나 생성된 문서의 품질 — 맞춤법, 공문서 규격 준수, 논리 흐름, 필수 항목 누락 여부 — 을 사용자가 직접 육안으로 확인해야 하는 상태다. 이는 다음 문제를 유발한다.

- 공문서 규격 미준수로 인한 반려/재작성 비용
- 맞춤법·어문 오류 누락으로 인한 신뢰도 저하
- 항목 누락 시 결재 흐름 중단

AI 문서 품질 검수 기능은 이 검토 부담을 AI에게 위임하여, 사용자가 문서 내용에만 집중할 수 있게 한다.

### 1-2. 사용자 가치

| 대상 | 현재 불편 | 이 기능 도입 후 |
|------|-----------|----------------|
| 문서 작성자 | 직접 맞춤법/규격 확인 필요 | 검수 결과 패널에서 즉시 확인 |
| 결재자 | 규격 오류 문서 반려 후 재검토 | 사전 필터링으로 결재 속도 향상 |
| 관리자 | 품질 편차 파악 어려움 | 검수 점수 기반 품질 추세 파악 가능 |

### 1-3. 관련 문서

- 후속 작업 계획서: `docs/01-plan/features/clio-next-phase.plan.md`
- 기존 문서 생성 API: `src/app/api/generate/route.ts`

## 4. 검수 항목 정의

GPT-4o에게 "한국 공문서 전문 교정자" 역할을 부여하고, 아래 4개 항목을 구조화된 JSON 형식으로 반환받는다.

### 4-1. 맞춤법 및 어문 규범

**검사 내용**
- 한글 맞춤법 위반 (띄어쓰기, 된소리, 사이시옷 등)
- 외래어 표기법 오류
- 문장 부호 오용 (쌍점, 마침표 위치 등)
- 높임말/평어 혼용

**GPT-4o 역할 지시 예시**
```
한국어 맞춤법 및 국립국어원 표준어 규정에 따라 오류를 찾아라.
위반 시 원문 인용 + 수정 제안을 제공하라.
```

### 4-2. 공문서 규격 준수

**검사 내용**
- 행정업무운영규정 기반 공문서 양식 준수 여부
- 수신/경유/발신 기재 형식
- 제목 표기 방식 (끝마침 없음 등)
- 날짜/금액 표기 형식 (예: 2026. 4. 12., 금 1,000,000원정)
- 문단 번호 체계 (1. → 가. → 1) → 가) 순서)

**GPT-4o 역할 지시 예시**
```
대한민국 행정안전부 공문서 작성 기준에 따라 규격 위반 항목을 검토하라.
```

### 4-3. 논리 흐름 및 문장 품질

**검사 내용**
- 문단 간 논리 연결 자연스러움
- 중복 표현 및 불필요한 수식어
- 능동/수동 표현 일관성
- 문장 길이 과도 (3줄 초과 단일 문장)
- 결론 없는 열거 구조

**GPT-4o 역할 지시 예시**
```
공문서 독자(결재자) 관점에서 논리 흐름이 명확한지, 핵심 내용이 먼저 오는지 검토하라.
```

### 4-4. 누락 항목 검사

**검사 내용**
- 필수 필드 미기재 (수신기관, 발신일, 문서번호 등)
- 서명/날인 안내 누락
- 첨부 파일 언급은 있으나 목록 없음
- 금액 기재 후 산출근거 누락
- 참조 조항 인용 후 조항 내용 미첨부

**GPT-4o 역할 지시 예시**
```
공문서 유형(보고서/계획서/계약서 등)에 맞는 필수 항목이 모두 포함되었는지 확인하라.
```

## 5. API 설계 (개요)

### 5-1. 요청

```
POST /api/quality-check
Authorization: Supabase 세션 쿠키

Body: {
  document_id: string    // documents 테이블 PK
}
```

### 5-2. 처리 흐름

```
클라이언트 요청
  → 세션 인증 확인
  → documents 테이블에서 content(마크다운) 조회
  → GPT-4o 호출 (system: 교정자 역할, user: 문서 내용)
  → JSON 응답 파싱
  → document_quality_checks 테이블 저장
  → 클라이언트에 결과 반환
```

### 5-3. GPT-4o 응답 형식 (JSON)

```json
{
  "overall_score": 82,
  "items": [
    {
      "category": "spelling",
      "severity": "error",
      "original": "되었으므로써",
      "suggestion": "되었으므로",
      "description": "조사 '써'는 '으로써'(수단)에만 사용합니다."
    },
    {
      "category": "format",
      "severity": "warning",
      "original": "2026년 4월 12일",
      "suggestion": "2026. 4. 12.",
      "description": "공문서 날짜 표기는 마침표(.) 구분 방식을 사용합니다."
    }
  ],
  "summary": "맞춤법 2건, 규격 오류 1건, 누락 항목 1건이 발견되었습니다."
}
```

### 5-4. DB 스키마 (신규 테이블)

```sql
CREATE TABLE document_quality_checks (
  id          UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  document_id UUID REFERENCES documents(id) ON DELETE CASCADE,
  checked_by  UUID REFERENCES users(id),
  overall_score INTEGER,
  result_json JSONB,        -- GPT-4o 응답 전체
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

## 6. UI 설계 (개요)

### 6-1. 진입점

- 위치: `app/documents/[id]/page.tsx` — 문서 상세 화면
- 버튼: "AI 검수" (문서 액션 영역, 다운로드 버튼 옆)

### 6-2. 결과 패널 구성

```
┌─────────────────────────────────────────┐
│ AI 품질 검수 결과              점수: 82  │
├──────────────────────────────────────── │
│ [맞춤법] 2건 오류                       │
│   · "되었으므로써" → "되었으므로"       │
│                                         │
│ [공문서 규격] 1건 경고                  │
│   · 날짜 형식: "2026년 4월 12일"        │
│     → "2026. 4. 12." 으로 수정 권장    │
│                                         │
│ [논리 흐름] 제안 없음                   │
│                                         │
│ [누락 항목] 1건 경고                    │
│   · 첨부파일 목록 언급 없음             │
│                                         │
│                          [재검수]       │
└─────────────────────────────────────────┘
```

### 6-3. 심각도 색상 코드

| 심각도 | 의미 | 색상 |
|--------|------|------|
| error | 즉시 수정 필요 | 빨강 (`text-red-600`) |
| warning | 수정 권장 | 노랑 (`text-yellow-600`) |
| suggestion | 참고 제안 | 파랑 (`text-blue-600`) |

## 7. 우선순위 (MoSCoW)

| 우선순위 | 항목 | 이유 |

| **Must (P0)** | 검수 버튼 + API 라우트 + 4종 항목 분석 + 결과 패널 UI | 핵심 가치 전달에 필수 |

## 9. 검증 기준

### 9-1. 기능 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| 맞춤법 오류 감지 | 의도적 오류 포함 문서로 검수 요청 | 오류 위치 + 수정 제안 반환 |
| 공문서 규격 감지 | 날짜를 "2026년 4월 12일" 형식으로 작성 후 검수 | 규격 위반 경고 반환 |
| 누락 항목 감지 | 수신기관 없는 공문 초안으로 검수 | 누락 항목 경고 반환 |
| 이미 검수된 문서 | 검수 후 페이지 재접속 | DB에서 결과 즉시 로드 (API 재호출 없음) |
| 재검수 | 문서 수정 후 재검수 클릭 | 새 결과로 갱신 |
| 비인증 접근 | 로그아웃 상태에서 API 직접 호출 | 401 Unauthorized |
| 응답 시간 | 5,000자 문서 기준 검수 요청 | 10초 이내 결과 반환 |

### 9-2. 회귀 테스트

| 항목 | 확인 방법 |
|------|-----------|
| 기존 문서 생성 흐름 | AI 검수 추가 후 HWPX/DOCX 생성 정상 동작 |
| 결재 워크플로우 | 검수 기능 추가 후 결재 요청/승인 흐름 정상 |
| documents 테이블 조회 | quality_checks JOIN 시 기존 목록 페이지 성능 영향 없음 |
