# CLIO 계약/문서 만료일 자동 알림 계획서

> **요약**: AI가 문서에서 만료일을 자동 추출하여 캘린더에 등록하고, 앱 진입 시 D-30 이내 만료 문서를 모달 팝업으로 즉시 알려주는 기능
>
> **프로젝트**: CLIO (RAG 기반 AI 문서관리 시스템)
> **버전**: v5.0.0
> **작성일**: 2026-04-12
> **상태**: Draft

---

## 1. 개요

### 1.1 목적 및 사용자 가치

기업 내부에서 관리하는 계약서, 유지보수 협약서, 보안서약서, 소프트웨어 라이선스 등
다양한 문서에는 만료일이 명시되어 있다. 현재 CLIO에는 문서 업로드 및 AI 분석 기능이 있지만,
만료일을 자동으로 감지하거나 기한이 임박했을 때 담당자에게 알려주는 기능이 없다.

담당자가 수백 건의 계약서를 수동으로 점검하다 보면 만료일을 놓쳐 자동 갱신이 실패하거나
서비스 중단 등의 리스크가 발생한다. 이 기능은 AI가 문서를 읽고 만료일을 자동으로 파악해
캘린더에 등록하고, D-30 / D-7 / D-1 시점에 알림을 발송함으로써 계약 관리 누락을
구조적으로 방지한다.

**핵심 사용자 가치**:
- 계약 만료 누락 리스크 제거
- 수동 일정 등록 작업 제거 (자동화)
- 앱 접속 즉시 만료 임박 문서를 모달로 확인 (별도 이메일/알림 설정 불필요)
- 만료 임박 문서를 대시보드에서 한눈에 파악
- 외부 서비스(이메일, cron) 없이 순수 클라이언트 사이드로 구현 가능

### 1.2 배경 및 맥락

CLIO는 현재 아래 인프라가 이미 실운영 중이다:
- **파일 텍스트 추출**: 업로드된 파일(PDF, DOCX, HWPX 등)에서 텍스트를 추출하는 파이프라인 완성
- **AI (GPT-4o)**: 문서 요약, 시맨틱 검색, 문서 생성에 이미 사용 중
- **schedules 테이블**: 일정/캘린더 기능이 이미 구현되어 있으며 `id, user_id, title, description, start_date, end_date, created_at` 컬럼 보유
- **pgvector 임베딩**: 문서 청크가 벡터로 저장되어 있어 내용 기반 검색 가능
- **Supabase Auth**: 사용자/부서 체계 및 RBAC 완성

추가 인프라 없이 기존 스택(Next.js 16, Supabase, GPT-4o, TypeScript) 위에서 구현 가능하다.

## 2. 구현 범위

### 2.1 In Scope (이번 구현 대상)

- [ ] AI가 문서 텍스트에서 만료일/계약기간 자동 추출 (GPT-4o)
- [ ] 추출된 만료일을 `schedules` 테이블에 자동 등록
- [ ] 앱 진입 시 D-30 이내 만료 문서 존재하면 모달 팝업 표시
- [ ] 모달에 "오늘 다시 보지 않기" 옵션 (localStorage 기반)
- [ ] 대시보드에 "만료 임박 문서" 위젯 표시 (D-30 이내)
- [ ] 파일 업로드 시 자동 트리거 (기존 업로드 파이프라인에 후처리로 추가)
- [ ] 만료일 수동 수정 UI (AI 추출 결과가 틀렸을 때 담당자가 직접 수정 가능)
- [ ] 추출 실패 시 "만료일 없음 / 추출 불가" 상태 명시

### 2.2 Out of Scope (이번 범위 제외)

- 이메일/SMS/푸시 알림 (외부 서비스 의존도 높음, 추후 검토)
- Slack / Teams / 카카오 등 외부 메신저 알림
- Cron Job / Edge Function 기반 백그라운드 알림
- 자동 계약 갱신 액션
- 여러 문서의 만료일 일괄 수정 UI
- 만료일 기반 문서 자동 아카이브/삭제
- 외부 캘린더(Google Calendar, Outlook) 연동

## 3. 만료일 감지 대상 문서 유형

AI 추출 시 아래 문서 유형을 우선 대상으로 한다.

| 시스템구축계약서 | 납품기한, 하자보증기간 종료일 | "납품일", "계약기간", "하자담보" |

## 6. 모달 팝업 방식 상세

### 6.1 팝업 트리거 조건

| 조건 | 설명 |
|------|------|
| 트리거 시점 | 로그인 후 메인 레이아웃 마운트 시 (앱 진입 시마다 체크) |
| 표시 조건 | D-30 이내 만료 문서가 1건 이상 존재할 때 |
| 재표시 방지 | localStorage `expiry_modal_suppressed_date` = 오늘 날짜이면 표시 안 함 |
| 표시 우선순위 | D-day 오름차순 (가장 임박한 것 먼저) |

### 6.2 모달 UI 구성

```
┌─────────────────────────────────────────┐
│  ⚠️ 만료 임박 문서 알림                  │
├─────────────────────────────────────────┤
│  D-30 이내 만료 예정인 문서가 있습니다.  │
│                                         │
│  · [D-3]  ABC사 유지보수계약서  2026.04.15 │
│  · [D-12] XYZ 소프트웨어 라이선스  2026.04.24 │
│  · [D-28] 보안서약서(홍길동)   2026.05.10 │
│                                         │
│  [오늘 다시 보지 않기]       [확인]      │
└─────────────────────────────────────────┘
```

### 6.3 "오늘 다시 보지 않기" 동작

- 클릭 시 `localStorage.setItem('expiry_modal_suppressed_date', 'YYYY-MM-DD')` 저장
- 앱 진입 시 저장된 날짜 = 오늘 날짜이면 모달 표시 생략
- 자정 이후 (다음 날 접속) 자동으로 다시 표시됨

## 7. 기술 설계 방향

### 7.1 DB 변경 사항

**schedules 테이블 — 컬럼 추가**
```sql
ALTER TABLE schedules
  ADD COLUMN IF NOT EXISTS source_type TEXT,       -- 'document_expiry' | NULL
  ADD COLUMN IF NOT EXISTS source_id UUID,          -- files.id 참조
  ADD COLUMN IF NOT EXISTS expiry_confidence TEXT;  -- 'high' | 'low' | 'none'
```

**별도 notifications 테이블 불필요** — 모달 팝업은 클라이언트에서 `/api/dashboard/expiry-summary` 응답을 직접 소비하므로 별도 알림 저장 테이블이 필요 없다. "오늘 다시 보지 않기" 상태는 localStorage에서 관리한다.

### 7.2 API 설계

| 엔드포인트 | 메서드 | 설명 |
|-----------|--------|------|
| `/api/files/[id]/extract-expiry` | POST | 특정 파일의 만료일 AI 추출 및 schedules 등록 |
| `/api/dashboard/expiry-summary` | GET | D-30 이내 만료 문서 목록 (모달 + 위젯 공용) |
| `/api/files/[id]/expiry` | PATCH | 만료일 수동 수정 |

### 7.3 클라이언트 모달 로직

메인 레이아웃 (`app/(main)/layout.tsx`) 또는 별도 `ExpiryAlertProvider` 컴포넌트에서:

```typescript
// 앱 진입 시 모달 표시 여부 결정
const today = new Date().toISOString().split('T')[0];
const suppressed = localStorage.getItem('expiry_modal_suppressed_date');
if (suppressed !== today) {
  // GET /api/dashboard/expiry-summary 호출
  // D-30 이내 문서 있으면 ExpiryAlertModal 표시
}
```

### 7.4 AI 추출 프롬프트 설계 방향

GPT-4o에게 문서 텍스트를 전달하고 아래 JSON 스키마로 응답을 요청한다:

```json
{
  "expiry_date": "YYYY-MM-DD",       // 만료일 (없으면 null)
  "contract_period": "YYYY-MM-DD ~ YYYY-MM-DD",  // 계약기간 (없으면 null)
  "document_type": "시스템구축계약서",  // 감지된 문서 유형
  "confidence": "high | low | none",   // 추출 신뢰도
  "reason": "추출 근거 또는 실패 이유"
}
```

토큰 절감을 위해 전체 텍스트가 아닌 앞 2000토큰만 사용하거나
기존 청크 중 날짜 관련 청크만 선별하여 전달한다.

### 7.5 업로드 파이프라인 연동

기존 파일 업로드 후처리 흐름에 비동기 단계로 추가한다:
```
파일 업로드 → 텍스트 추출 → 청킹 → 임베딩 → [만료일 추출 → schedules 등록 → 알림 생성]
```
만료일 추출 실패가 전체 업로드를 막지 않도록 try/catch로 격리한다.

## 8. 검증 기준 (Acceptance Criteria)

### 8.1 AI 추출 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| 계약서 만료일 추출 | 명시적 날짜가 있는 계약서 PDF 업로드 | 만료일 정상 파싱, schedules 등록 확인 |
| 날짜 없는 문서 | 날짜 미포함 문서 업로드 | `confidence: none`, 에러 없이 "만료일 없음" 상태 |
| 추출 결과 오류 수정 | 잘못된 날짜 → 수동 수정 | schedules 레코드 업데이트, 알림 일정 재계산 |

### 8.2 캘린더 등록 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| schedules 자동 등록 | 파일 업로드 후 일정 페이지 확인 | 만료일 기준 일정 자동 생성 |
| source_type 확인 | DB에서 schedules 레코드 조회 | `source_type = 'document_expiry'` 확인 |
| RLS 격리 | 다른 계정으로 로그인 후 일정 조회 | 본인 일정만 노출 |

### 8.3 모달 팝업 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| 모달 표시 | D-30 이내 만료 파일 등록 후 앱 재진입 | 만료 임박 모달 팝업 표시 |
| 모달 미표시 | 만료일 없거나 D-31 이상 파일만 있을 때 앱 진입 | 모달 미표시 |
| "오늘 다시 보지 않기" | 버튼 클릭 후 앱 재진입 | 당일 모달 미표시 |
| 익일 재표시 | localStorage 날짜를 어제로 변경 후 앱 진입 | 모달 재표시 |

### 8.4 대시보드 위젯 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| 위젯 표시 | 만료일 D-30 이내 파일 존재 시 대시보드 로드 | "만료 임박 문서" 위젯 정상 렌더링 |
| D-30 초과 제외 | 만료까지 31일 이상 남은 문서 | 위젯에 미표시 |
| 만료됨 구분 | 만료일이 오늘 이전인 문서 | "만료됨" 배지 또는 별도 구분 표시 |

### 8.5 회귀 테스트

알림 기능 추가 후 아래 기존 기능이 정상 동작하는지 확인한다:

| 항목 | 확인 방법 |
|------|-----------|
| 파일 업로드 전체 파이프라인 | 업로드 → 임베딩 완료 상태 정상 |
| 캘린더 수동 일정 등록 | 기존 일정 CRUD 정상 |
| 시맨틱 검색 | AI 검색 결과 정상 |
| 대시보드 기존 위젯 | 기존 통계 정상 표시 |
