## 1. 문서 목적

이 문서는 현재 CLIO의 실제 코드 구조와 서비스 분석 결과를 바탕으로,  
제품 방향을 재정렬하고 구현 우선순위를 명확히 하기 위한 실행 문서다.

특히 Claude Code가 다음을 바로 이해하고 작업할 수 있도록 작성한다.

- 지금 CLIO의 핵심 가치가 무엇인지
- 무엇을 먼저 고쳐야 하는지
- 어떤 화면과 파일부터 손대야 하는지
- 각 단계의 완료 기준이 무엇인지

## 2. 현재 진단 요약

### 2-1. 현재 제품의 강점

CLIO는 이미 다음 기능을 실운영 수준으로 갖추고 있다.

- 파일 업로드 및 문서 파싱
- 벡터 임베딩 기반 AI 검색
- 멀티 포맷 문서 생성
- 문서 댓글 및 AI 반영
- 계약서 리스크 분석 및 수정 제안
- STT 회의록 및 할일 추출
- 메모, 메시지, 일정, 사용자/부서 관리

즉, 기능 부족보다 **기능 연결 방식과 제품 메시지 정렬**이 문제다.

### 2-2. 핵심 문제

현재 CLIO는 기능이 많지만 사용자가 처음 진입했을 때 핵심 가치가 분산되어 보인다.

주요 문제는 다음과 같다.

1. 메뉴가 병렬적이라 제품 중심축이 약하다.
2. `파일 관리`, `AI 검색`, `문서 생성`, `계약 리스크`가 서로 이어지지 않는다.
3. 사용자가 파일 업로드 후 “다음에 무엇을 해야 하는지” 시스템이 안내하지 않는다.
4. 강력한 기능인 `계약 리스크`가 파일 허브와 분리되어 있다.
5. 메시지/일정/메모 같은 부가 기능이 메인 가치와 같은 레벨에 노출된다.

### 2-3. 개선의 기본 원칙

CLIO는 앞으로 다음 한 문장으로 정리되어야 한다.

> CLIO는 기업 문서를 업로드하고, 찾고, 검토하고, 생성하고, 계약 리스크까지 처리하는 AI 문서 업무 플랫폼이다.

즉, 개선 방향은 **문서 중심 허브화**다.

## 3. 목표 상태

### 3-1. 제품 포지셔닝

CLIO의 중심 제품 정의는 아래로 고정한다.

- 1순위: AI 문서관리
- 2순위: 문서 기반 업무 자동화
- 3순위: 계약 검토 특화 기능

### 3-2. 핵심 사용자 흐름

개선 후 핵심 흐름은 아래와 같아야 한다.

1. 사용자가 파일을 업로드한다.
2. CLIO가 텍스트 추출, 분류, 요약, 만료일, 계약 여부를 자동 처리한다.
3. 시스템이 다음 행동을 추천한다.
4. 사용자는 아래 중 하나로 즉시 이동한다.
   - AI 검색
   - 문서 생성
   - 계약 리스크 분석
   - 공유/협업

### 3-3. 메뉴 구조 목표

최종적으로 메뉴 구조는 아래 방향을 지향한다.

- 문서 허브
  - 대시보드
  - 파일 관리
  - AI 검색
  - 문서 생성
- 고급 분석
  - 계약 리스크
  - 메모
- 협업
  - 메시지
  - 일정/할일
- 관리
  - 설정

사이드바에서 반드시 대분류를 나누라는 뜻은 아니다.  
핵심은 사용자가 메뉴를 봤을 때 **문서가 중심**으로 느껴지게 만드는 것이다.

### 포함 범위

- 정보구조(IA) 재편

- 계약 리스크의 파일 허브 연동

### 제외 범위

- DB 전면 개편

- 메시지/일정의 대규모 기능 추가
- 메모 기능 전면 재설계

## 5. 단계별 실행 로드맵

## Phase 1. 제품 중심축 정리

### 목표

메뉴와 대시보드에서 CLIO의 중심이 문서라는 점을 명확히 보이게 만든다.

### 작업 항목

1. 사이드바 메뉴를 그룹 관점에서 재정리한다.
2. 대시보드의 빠른 액션과 주요 위젯을 문서 흐름 중심으로 재배치한다.
3. `템플릿 관리`, `계약 리스크`, `AI 검색`의 관계를 대시보드에서 설명한다.
4. 메시지/일정/메모는 보조 기능으로 보이도록 시각적 우선순위를 낮춘다.

### 우선 수정 파일

- `src/components/layout/sidebar.tsx`
- `src/app/(app)/dashboard/page.tsx`
- `src/lib/constants/ui.ts`

### 완료 기준

- 첫 화면에서 사용자가 CLIO를 `문서 업무용 제품`으로 인식할 수 있다.
- 대시보드의 1차 CTA가 파일 업로드, AI 검색, 문서 생성, 계약 리스크로 정렬된다.
- 보조 기능이 메인 기능과 동급으로 보이지 않는다.

---

## Phase 2. 파일 관리 화면을 문서 허브로 전환

### 목표

파일 관리 화면을 단순 보관소가 아니라, 모든 후속 작업의 출발점으로 만든다.

### 작업 항목

1. 파일 목록/상세에서 아래 정보가 바로 보이도록 확장한다.
   - 미리보기
   - 요약/핵심 정보
   - 파일 유형
   - 계약서 후보 여부
   - 만료일
   - 접근 범위
2. 파일별 추천 액션 UI를 추가한다.
   - `AI 검색에 포함`
   - `문서 생성에 사용`
   - `계약 리스크 분석`
   - `공유`
3. 파일 상세 모달 또는 우측 패널을 도입해 후속 액션을 한곳에 모은다.
4. 업로드 완료 직후 “다음 할 일” 안내를 보여준다.

### 우선 수정 파일

- `src/app/(app)/files/page.tsx`
- `src/app/api/files/route.ts`
- `src/app/api/files/[id]/preview/route.ts`
- 필요 시 `src/components/ui/*`

### 완료 기준

- 사용자가 파일을 업로드한 뒤 다른 메뉴로 헤매지 않고 바로 다음 행동을 선택할 수 있다.
- 계약서/일반문서/회의자료 등 파일 유형에 따라 적절한 액션이 노출된다.
- 파일 화면 하나만으로도 CLIO의 주요 가치가 드러난다.

---

## Phase 3. AI 검색을 메인 탐색 경험으로 강화

### 목표

검색을 단순 조회가 아니라, 답변과 실행으로 이어지는 중심 경험으로 만든다.

### 작업 항목

1. 검색 결과 화면에서 `답변 + 근거 파일 + 다음 액션` 구조를 강화한다.
2. 검색 결과에서 바로 문서 생성, 공유, 계약 분석으로 이동할 수 있게 한다.
3. 파일 검색과 AI Q&A 탭의 연결을 더 명확하게 만든다.
4. “이 결과로 무엇을 할 수 있는지” 액션 버튼을 표준화한다.

### 우선 수정 파일

- `src/app/(app)/search/page.tsx`
- `src/app/api/search/route.ts`
- `src/app/api/chat/route.ts`

### 완료 기준

- 검색 결과가 단순 리스트가 아니라 업무 행동으로 이어진다.
- 사용자가 검색 후 별도 메뉴 탐색 없이 문서 생성 또는 분석 작업을 수행할 수 있다.

---

## Phase 4. 문서 생성과 검토 흐름 통합

### 목표

문서 작성, 검토, 반영, 비교, 품질 점검이 하나의 작업 흐름처럼 느껴지게 만든다.

### 작업 항목

1. `문서 생성` 화면에서 소스 파일 선택 흐름을 더 명확하게 정리한다.
2. 생성된 문서의 다음 단계가 명확히 보이게 만든다.
   - 댓글 검토
   - AI 반영
   - 품질 점검
   - 버전 비교
   - 다운로드
3. 완료 문서와 초안 문서의 상태 차이를 UI에서 더 분명히 표현한다.
4. 문서 뷰어에서 후속 작업 버튼 우선순위를 재정렬한다.

### 우선 수정 파일

- `src/app/(app)/documents/page.tsx`
- `src/app/(app)/documents/[id]/page.tsx`
- `src/components/documents/*`
- `src/components/document-diff/*`

### 완료 기준

- 사용자가 `생성 -> 검토 -> 반영 -> 완료` 흐름을 자연스럽게 따라갈 수 있다.
- 문서 생성 이후 다음 작업이 숨지 않고 명시적으로 노출된다.

---

## Phase 5. 계약 리스크를 파일 허브에 연결

### 목표

가장 전문성이 강한 기능인 계약 리스크를 독립 기능이 아니라 문서 업무 흐름 안에 연결한다.

### 작업 항목

1. 파일 관리 화면에서 계약서 파일을 바로 계약 리스크 분석으로 연결한다.
2. 기존 파일을 재업로드하지 않고 분석할 수 있는 흐름을 검토한다.
3. 계약 리스크 결과 화면에서 아래 후속 액션을 강화한다.
   - 수정 제안 받기
   - 수정본 다운로드
   - 관련 파일과 연결
   - 담당자 공유
4. 대시보드나 파일 허브에서 “최근 계약 분석” 상태를 요약 노출한다.

### 우선 수정 파일

- `src/app/(app)/contract-risk/page.tsx`
- `src/app/(app)/contract-risk/[id]/page.tsx`
- `src/app/api/contract-risk/analyze/route.ts`
- `src/app/(app)/files/page.tsx`

### 완료 기준

- 계약 검토 기능이 별도 고립 기능이 아니라 CLIO 핵심 가치의 일부로 보인다.
- 사용자가 파일 보관과 계약 분석을 따로 인식하지 않는다.

## 6. 추천 구현 순서

Claude Code는 아래 순서로 작업하는 것이 가장 안전하다.

1. Phase 1: 사이드바/대시보드 개편
2. Phase 2: 파일 허브화
3. Phase 3: 검색 액션 연결
4. Phase 4: 문서 생성/검토 흐름 정리
5. Phase 5: 계약 리스크 연동

이 순서를 권장하는 이유:

- 정보구조를 먼저 정리해야 이후 UI 수정이 흔들리지 않는다.
- 파일 허브가 정리되어야 검색/문서/계약 기능 연결 포인트가 생긴다.
- 계약 리스크는 강력하지만 메인 허브 정리 후 붙이는 편이 전체 흐름이 자연스럽다.

## 7. Claude Code 작업 지침

### 7-1. 작업 원칙

Claude Code는 아래 원칙으로 구현한다.

1. 새 기능을 늘리기보다 기존 기능 간 연결을 우선한다.
2. 한 번에 전면 개편하지 말고 화면 단위로 안전하게 나눈다.
3. API 변경이 필요한 경우 UI와 함께 최소 범위로 수정한다.
4. 기존 권한 모델과 데이터 구조를 최대한 유지한다.

### 7-2. 화면 단위 작업 방식

각 단계는 아래 형식으로 진행한다.

1. 현재 화면 구조 파악
2. 사용자 목표 재정의
3. CTA 우선순위 조정
4. 필요 시 API 응답 필드 보강
5. UI 반영
6. 수동 검증

### 7-3. 절대 피해야 할 것

- 메시지/일정/메모를 이번 작업의 중심으로 끌어올리는 것
- 새 대형 기능을 추가하다가 핵심 흐름 정비를 미루는 것
- 파일 허브 없이 문서 생성/계약 기능만 따로 고도화하는 것
- 메뉴 이름만 바꾸고 실제 흐름 연결을 하지 않는 것

## 9. 검증 기준

### UX 검증 질문

아래 질문에 “예”라고 답할 수 있어야 한다.

1. 첫 화면만 보고도 CLIO가 문서 업무 플랫폼인지 이해되는가?

- 파일 업로드 후 다른 핵심 기능 진입률

- 문서 생성 후 완료 문서 전환률

## 11. 최종 결론

이번 개선의 본질은 기능 추가가 아니다.  
CLIO를 **문서 중심 AI 업무 플랫폼**으로 다시 정렬하는 작업이다.

가장 중요한 구현 포인트는 아래 세 가지다.

1. 문서가 제품의 중심이라는 점을 UI에서 명확히 보이게 만들 것
2. 파일 업로드 이후 다음 행동을 시스템이 안내할 것
3. 검색, 문서 생성, 계약 리스크를 하나의 흐름으로 연결할 것

Claude Code는 이 문서를 기준으로, 화면 단위로 안전하게 개선을 진행하면 된다.
