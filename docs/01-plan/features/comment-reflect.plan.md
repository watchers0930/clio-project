# 계획서: 댓글 반영 고도화 (comment-reflect)

**작성일:** 2026-04-13  
**버전:** v1.0  
**상태:** Plan

---

## 1. 배경 및 목적

현재 댓글 반영 기능은 선택된 댓글을 AI에 전달해 문서 전체를 재작성하는 방식이다.  
이 방식은 원문 구조를 훼손하고, 불필요한 스냅샷 문서가 DB에 누적되는 문제가 있다.

**목표:** 원문은 보존하면서, 댓글을 두 가지 방식으로 원문에 직접 합치는 기능 구현

---

## 2. 기능 정의

### 모드 1 — 기존 섹션에 삽입

- 사용자가 댓글을 선택한 후, 원문의 어느 섹션에 넣을지 선택
- 선택된 댓글 내용이 해당 섹션 안에 추가됨
- AI가 댓글을 해당 섹션 문체에 맞게 자연스럽게 녹여 삽입

### 모드 2 — 새 단락 생성

- 사용자가 댓글을 선택한 후, 새 단락 이름을 입력 (예: 해결방안, 조치사항, 의견 종합)
- AI가 선택된 댓글들을 정리해 단락 내용 생성
- 원문 마지막 아래에 새 단락으로 추가

### 공통

- 두 모드 모두 **원문 하나에 합쳐진 형태**로 저장
- 새 문서 레코드(스냅샷) 생성 없음
- 반영 후 뷰어에서 즉시 확인 가능

---

## 3. 사용자 흐름

```
댓글 패널 → 댓글 체크박스 선택
         → "반영하기" 클릭
         → 팝업: [기존 섹션에 추가] vs [새 단락 만들기]

모드 1 선택:
  → 원문 섹션 목록 표시 (정보출처 / 보고내용과 의견 / 문제점 등)
  → 섹션 선택 → 적용 → 해당 섹션에 내용 추가

모드 2 선택:
  → 새 단락 이름 입력 (예: 해결방안)
  → AI 생성 → 원문 하단에 새 단락 추가
```

---

## 4. 현재 구조와의 차이점

| 항목 | 현재 | 변경 후 |
|------|------|---------|
| AI 처리 범위 | 문서 전체 재작성 | 선택 섹션 or 새 단락만 |
| DB 변화 | 스냅샷 문서 새로 생성 | 현재 문서 content만 업데이트 |
| 원문 보존 | 스냅샷으로 분리 | 원문 내에 합쳐서 보존 |
| 버전 관리 | parent_id 체인 | 변경 없음 (단순 content update) |

---

## 5. 범위 (Scope)

### In Scope
- 댓글 반영 팝업 UI (모드 선택)
- 모드 1: 섹션 선택 → AI 삽입 API
- 모드 2: 단락명 입력 → AI 생성 → 문서 하단 추가 API
- 기존 reflect API 교체 또는 병행

### Out of Scope
- 반영 이력 추적 (어떤 댓글이 반영됐는지)
- 반영 취소/롤백
- HWPX 파일 자체 수정

---

## 6. 우선순위

| 항목 | 우선순위 |
|------|---------|
| 모드 2 (새 단락) | ★★★ 높음 — 단순, 즉시 가치 |
| 모드 1 (섹션 삽입) | ★★☆ 중간 — 섹션 파싱 필요 |
| UI 팝업 | ★★★ 높음 — 두 모드 진입점 |

---

## 7. 기술 고려사항

- 원문이 HWPX 기반이면 `content` 필드에 마크다운 추가 형태로 보완
- 섹션 파싱: 현재 content의 `##`, `###` 헤더 기준으로 섹션 분리
- API: 기존 `/reflect`를 확장하거나 별도 `/apply-comments` 엔드포인트 신설
- 반영 후 `fetchDoc()` 재호출로 뷰어 갱신
