## 1. 맥락기술서 (Context Document)

### 1-1. 기능 목적

CLIO는 기업 내부 문서를 AI로 생성하고 관리하는 시스템이다.  
문서가 개정될수록 "이번 버전에서 무엇이, 왜 바뀌었는가"를 빠르게 파악하는 것이  
계약서/보고서 업무에서 핵심 가치다.

**Doc Diff** 기능은 두 버전의 문서를 나란히 펼쳐 변경된 텍스트를 시각적으로  
표시하고, GPT-4o가 변경 맥락(유불리, 법적 함의, 주요 수정 이유)을 자연어로 해석해 주는  
AI 강화 비교 뷰어다.

### 1-2. 사용자 가치

| 사용자 유형 | 기존 불편 | 이 기능으로 해결 |
|------------|-----------|-----------------|
| 법무/계약 담당자 | 두 계약서 버전을 수작업 비교 | 변경 조항 즉시 시각화 + "갑에게 불리한 조항" AI 해석 |
| 팀장/결재자 | 개정 이유를 기안자에게 직접 물어야 함 | AI 요약으로 변경 맥락 즉시 파악 |
| 일반 직원 | 이전 버전 어디에 무엇이 있었는지 기억 의존 | 버전 체인으로 히스토리 추적 |

### 1-3. 기존 구현과의 연계

- `documents` 테이블: `parent_id`, `version_number` 컬럼 이미 구현 (P3-1)
- `/api/documents/[id]/versions/route.ts`: 버전 목록 조회 API 이미 구현
- AI: OpenAI GPT-4o (`/api/generate/route.ts`에서 사용 중)
- 스택: TypeScript + Next.js 16 + Supabase

### 1-4. 관련 문서

- 참고 계획서: `docs/01-plan/features/clio-next-phase.plan.md`
- 연계 기능: P3-1 문서 버전 관리 (parent_id 체인)

## 2. 범위 정의 (Scope)

### 2-1. In Scope (이번 구현)

- [ ] 동일 문서의 두 버전 선택 UI (드롭다운 or 타임라인)
- [ ] 사이드바이사이드(side-by-side) 비교 뷰 — 기본 모드
- [ ] 인라인(inline) 비교 뷰 — 토글 전환
- [ ] 라인 단위 diff 계산 (추가/삭제/변경 색상 강조)
- [ ] 단어/문자 단위 세부 하이라이트 (changed word 내 세밀 표시)
- [ ] AI 변경 해석 패널 (어떤 내용이 왜 바뀌었는지 + 유불리 분석)
- [ ] 계약서 특화 AI 해석 ("갑/을에게 유리/불리한 조항" 분류)
- [ ] 변경 항목 네비게이션 (이전/다음 변경으로 점프)
- [ ] 현재 버전과 최신 버전 비교 (Quick Compare)
- [ ] 비교 결과 공유 링크 (읽기 전용 URL)

### 2-2. Out of Scope (이번 미구현)

- 3개 이상 버전 동시 비교
- PDF/DOCX/HWPX 원본 파일 바이너리 비교 (텍스트 콘텐츠만 비교)
- 외부 문서(CLIO 외부에서 생성된 파일) 비교
- 변경 이력 승인/거절 워크플로우 연동
- 실시간 공동 편집 병합(merge) 기능

## 5. diff 표시 방식

### 5-1. 알고리즘

- **Myers diff 알고리즘** 적용 (라인 단위)
- 라이브러리: `diff` (npm, 경량) 또는 `jsdiff`
- 단어 단위 세부 diff: Myers 결과를 다시 단어 토큰으로 재분석

### 5-2. 색상 체계

| 변경 유형 | 배경 | 텍스트 | 아이콘 |
|-----------|------|--------|--------|
| 추가 (Added) | `#DCFCE7` (연초록) | `#166534` | `+` |
| 삭제 (Removed) | `#FEE2E2` (연빨강) | `#991B1B` | `-` |
| 수정 (Changed) | `#FEF9C3` (연노랑) | `#713F12` | `~` |
| 동일 (Unchanged) | 기본 배경 | 기본 텍스트 | — |

색약 대응: 색상 외에 좌측 거터(gutter)에 `+` / `-` / `~` 기호 병용

### 5-3. 뷰 모드

#### Side-by-Side (기본)

```
┌─────────────────────┬─────────────────────┐
│  v1.0 (2026-03-01)  │  v2.0 (2026-04-12)  │
├─────────────────────┼─────────────────────┤
│  1 | 계약 기간: 1년  │  1 | 계약 기간: 2년  │  ← 변경
│  2 | 대금: 1,000만원 │  2 | 대금: 1,200만원 │  ← 변경
│  3 | 지체보상금: 0.1% │  3 | 지체보상금: 0.1% │
│    |                 │  4 | 하자보증: 1년   │  ← 추가
└─────────────────────┴─────────────────────┘
```

#### Inline (토글)

```
1  ~~계약 기간: 1년~~ → 계약 기간: 2년
2  ~~대금: 1,000만원~~ → 대금: 1,200만원
3  지체보상금: 0.1%
4  [추가] 하자보증: 1년
```

### 5-4. 네비게이션 바

- 화면 우측 고정 미니맵 또는 상단 툴바
- "← 이전 변경" / "다음 변경 →" 버튼
- 변경 수 요약: `+ 3줄 추가  - 1줄 삭제  ~ 2줄 변경`

## 6. AI 해석 범위

### 6-1. 기본 AI 해석 (모든 문서 공통)

GPT-4o에 두 버전의 diff를 전달하여 아래 내용을 생성:

```
[변경 요약]
총 N개 항목이 변경되었습니다.

[주요 변경 내용]
1. 계약 기간: 1년 → 2년 (연장)
2. 대금: 1,000만원 → 1,200만원 (20% 인상)
3. 하자보증 조항 신설 (1년)

[변경 맥락]
이번 개정은 계약 기간 연장과 함께 대금이 상향 조정되었으며,
하자보증 조항이 새로 추가되어 공급자의 책임 범위가 확대되었습니다.
```

### 6-2. 계약서 특화 AI 해석

문서 타입이 계약서(`contract` 종류)인 경우 추가 분석 실행:

```
[유불리 분석]
갑(발주자)에게 유리한 변경:
  - 하자보증 조항 신설 → 공급자(을) 책임 강화

을(공급자)에게 유리한 변경:
  - 계약 기간 연장 → 납기 여유 확보
  - 대금 인상 → 수익 개선

중립적 변경:
  - 없음

[법적 주의 사항]
하자보증 기간 및 범위가 명확히 정의되지 않았습니다.
구체적인 하자 범위를 추가 협의하는 것을 권장합니다.
```

### 6-3. AI 호출 전략

- diff 텍스트 + 문서 메타(타입, 제목) → GPT-4o system prompt 구성
- **스트리밍 응답** (SSE): 사용자가 기다리는 동안 점진적 표시
- diff 없을 시 (두 버전 동일): AI 패널 숨김, "두 버전이 동일합니다" 표시
- AI 오류 시: diff 뷰는 유지, AI 패널에 fallback 메시지 표시

## 7. 신규 API 설계

### 7-1. 버전 비교 엔드포인트

```
POST /api/documents/[id]/diff
Body: { fromVersionId: string, toVersionId: string }
Response: {
  diff: DiffChunk[],
  stats: { added: number, removed: number, changed: number }
}
```

### 7-2. AI 해석 엔드포인트 (스트리밍)

```
POST /api/documents/[id]/diff/analyze
Body: { fromVersionId: string, toVersionId: string, documentType: string }
Response: SSE stream (text/event-stream)
```

### 7-3. RLS 보안

- 기존 `documents` 테이블 RLS: `auth.uid() = user_id OR organization_id 일치`
- diff API도 동일 RLS 정책 상속 — 타 조직 버전 접근 차단

## 8. 검증 기준 (Acceptance Criteria)

### 8-1. 기능 검증

| 검증 항목 | 방법 | 기대 결과 |
|-----------|------|-----------|
| diff 정확성 | 알려진 변경 문서 2개 버전 비교 | 추가/삭제 라인 100% 정확 표시 |
| 색상 구분 | 추가/삭제/변경/동일 항목 확인 | 4가지 색상 정확히 적용 |
| 사이드바이사이드 뷰 | 라인 번호 정렬 | 동일 라인 좌우 수평 정렬 |
| 인라인 뷰 전환 | 토글 버튼 클릭 | 300ms 이내 뷰 전환 |
| 변경 네비게이션 | 이전/다음 버튼 | 해당 diff 항목으로 스크롤 이동 |
| AI 해석 생성 | 변경된 문서 비교 후 AI 패널 | 30초 이내 전체 해석 완료 |
| 계약서 유불리 분석 | 계약서 타입 문서 비교 | 갑/을 유불리 섹션 표시 |
| AI 오류 fallback | AI API 키 제거 후 비교 | diff 뷰 정상, AI 패널 오류 메시지 |
| RLS 보안 | 타 사용자 토큰으로 API 직접 호출 | 403 또는 빈 결과 반환 |
| 동일 버전 비교 | 동일한 두 버전 ID 선택 | "변경 사항 없음" 표시 |

### 8-2. 성능 검증

| 항목 | 기준 | 도구 |
|------|------|------|
| diff 렌더링 속도 | 10,000자 기준 1초 이내 | Chrome DevTools Performance |
| AI 응답 TTFB | 3초 이내 첫 토큰 | Network 탭 |
| 메모리 사용량 | diff 뷰 오픈 후 100MB 이하 추가 | Chrome Memory 탭 |

### 8-3. 회귀 테스트

Doc Diff 기능 완료 후 아래 항목 반드시 재확인:

| 항목 | 확인 방법 |
|------|-----------|
| 문서 생성 정상 동작 | 기존 DOCX/HWPX 생성 후 다운로드 |
| 버전 목록 API | `GET /api/documents/[id]/versions` 응답 확인 |
| 결재 워크플로우 | 버전 관련 문서 결재 요청 → 승인 플로우 |
